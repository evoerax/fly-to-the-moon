# fttm v0.2.1 架构重构规格文档

**版本**: 0.2.1  
**目标**: 全面优化架构，解决记忆膨胀、观测性、配置系统等问题

---

## v0.1 vs v0.2.1 对比

| 特性               | v0.1                        | v0.2.1                                              |
| ------------------ | --------------------------- | --------------------------------------------------- |
| **记忆系统**       | notes.md 无限膨胀           | 滑动窗口 + 分层记忆                                 |
| **记忆注入**       | 全部历史注入 prompt         | 按需读取 + 近期完整注入                             |
| **长对话压缩**     | 无                          | 100+ 迭代后自动压缩 notes.md                        |
| **迭代结果缓存**   | 无                          | 相同 scope 缓存命中复用                             |
| **上下文拉满**     | 轻量 prompt，agent 读文件   | `--context-mode full` 把历史直接注入 context window |
| **配置**           | 仅全局 `~/.fttm/config.yml` | 分层配置（系统/项目/CLI）                           |
| **运行报告**       | 无                          | `report.json` 持久化                                |
| **Token 统计**     | 仅运行时显示                | `stats.jsonl` 持久化                                |
| **Checkpoint**     | 无                          | 迭代快照 + Crash 恢复                               |
| **CLI**            | 单命令                      | 子命令（start/resume/status/log/report/stop）       |
| **Hooks**          | 无                          | 迭代前后/提交前后钩子                               |
| **目录结构**       | `.fttm/runs/<id>/` 扁平     | `.fttm/runs/<id>/iterations/` 分层                  |
| **Agent 能力声明** | 无                          | `AgentMetadata` 接口                                |
| **Claude model**   | 不支持                      | `--model` 支持                                      |
| **统筹 Agent**     | 无                          | Coordinator 监督 + 危险操作确认                     |

### 破坏性变更

| 变更                     | v0.1     | v0.2.1                          |
| ------------------------ | -------- | ------------------------------- |
| `.fttm/` 目录            | 扁平结构 | 子目录 `iterations/`            |
| notes.md                 | 完整历史 | 仅近期 + 归档                   |
| CLI 子命令               | 无       | `fttm status`/`log`/`report` 等 |
| `maxConsecutiveFailures` | 硬编码 3 | 可配置                          |

## 1. 记忆与上下文系统

### 问题

- `notes.md` 无限膨胀，50+ 迭代后 context 消耗大
- 所有历史完整注入 prompt，导致 token 浪费

### 解决方案

#### 1.1 滑动窗口记忆

- 只保留最近 **N 次迭代**的完整内容（默认 N=10）
- 早期迭代合并成摘要
- 配置项：`maxRecentIterations: 10`

#### 1.2 分层记忆

| 层级              | 内容                               | 注入方式     |
| ----------------- | ---------------------------------- | ------------ |
| 近期（最近 N 次） | 完整 summary + changes + learnings | 注入 prompt  |
| 中期              | 仅 summary（压缩成一行）           | 注入 prompt  |
| 历史              | JSONL 原始文件                     | agent 自己读 |

#### 1.3 按需读取

- prompt 里告诉 agent："如需了解历史详情，读取 `.fttm/runs/<run-id>/iteration-K.jsonl`"
- agent 自己决定是否需要查历史，而非每次都注入

#### 1.4 长对话压缩（100+ 迭代场景）

**问题**：迭代超过 100 次后，即使用滑动窗口，notes.md 和 prompt 仍然很长，token 消耗巨大。

**解决方案**：

- 当 `notes.md` 超过配置阈值（默认 10KB）时，触发"压缩"
- 压缩时把所有 `key_changes_made` 合并成一个"累计变更列表"
- 所有 `key_learnings` 合并成一个"累计经验库"
- notes.md 重写为：`=== 压缩摘要 ===` + 累计变更 + 累计经验 + 最近 N 次迭代完整内容
- 压缩后 iteration 编号保留，但中间历史不再逐条展开

#### 1.5 上下文拉满模式

**问题**：

- 当前 iteration prompt 很轻量（几百字），没有充分利用模型的 context window
- notes.md 是 agent 自己读的，读的 token 消耗了但没注入到可用空间
- 缺乏"用尽可用 context"的设计

**解决方案**：

- 新增 `--context-mode full`（默认 `compact`）
- `full` 模式下：把 notes.md 内容直接注入 prompt，而不是告诉 agent "先读 notes.md"
- 计算可用 context budget（模型 context window - reserved space），把尽量多的历史记忆注入
- Prompt 结构变为：
  ```
  ## Objective
  ## Available Context Budget (~180K tokens)
  ## Recent History (fully injected, ~30K tokens)
  ## Current Task
  ## Output Schema
  ```
- `compact` 模式（默认）保持现有行为：轻量 prompt + agent 自己读历史

**问题**：如果目标很大但被拆成很多小 iteration，每次切 iteration 都要重新理解上下文，浪费 token。

**解决方案**：

- 对每次 iteration 的 `objective + scope + key_changes` 做 hash
- 如果下一轮 iteration 的 scope hash 与历史某次完全相同，且那次的 `success=true`，直接复用那次的结果
- 配置项：`cacheHitsEnabled: true`
- 同时支持手动 bypass：`--no-cache`

## 2. Agent 抽象层

### 问题

- Agent 接口虽已抽象，但扩展新 agent 仍需修改多处
- 不同 agent 的 token 计数逻辑分散

### 解决方案

#### 2.1 统一 Token 报告接口

```typescript
interface AgentResult {
  output: AgentOutput;
  usage: {
    inputTokens: number;
    outputTokens: number;
    cacheReadTokens?: number; // claude 特有
    cacheCreationTokens?: number; // claude 特有
  };
}
```

#### 2.2 Agent 注册表

```typescript
const AGENTS = {
  claude: ClaudeAgent,
  codex: CodexAgent,
  rovodev: RovoDevAgent,
  opencode: OpenCodeAgent,
} as const;
```

#### 2.3 Agent 能力声明

```typescript
interface AgentMetadata {
  name: string;
  supportsStreaming: boolean;
  supportsJsonSchema: boolean;
  supportsCustomModel: boolean;
}
```

#### 2.4 Claude Model 支持

目前只有 `opencode` agent 支持 `--model`，Claude 不支持。

v0.2.1 给 Claude 也加上：

```typescript
// claude.ts 修改
spawn(this.bin, [
  "-p",
  prompt,
  "--model",
  modelName, // 新增：支持指定模型
  "--verbose",
  "--output-format",
  "stream-json",
  "--json-schema",
  JSON.stringify(AGENT_OUTPUT_SCHEMA),
  "--dangerously-skip-permissions",
]);
```

```yaml
# 配置示例
agent: claude
model: sonnet # 或 opus、haiku 等
```

## 3. 状态管理与恢复

### 问题

- crash 后只能 resume，可能丢失最后一次迭代
- 没有手动 checkpoint

### 解决方案

#### 3.1 迭代快照

每次迭代开始前写入 `.fttm/runs/<run-id>/checkpoint.json`：

```json
{
  "iteration": 5,
  "startedAt": "ISO8601",
  "workingTreeHash": "sha"
}
```

#### 3.2 自动恢复

- 启动时检测 checkpoint
- 如果 checkpoint 显示上次未完成，提供选项：
  - 继续（retry last iteration）
  - 跳过（treat as failed）
  - 中止

#### 3.3 Crash 保护

- 捕获 SIGTERM/SIGINT 时先写 checkpoint 再退出
- 防止孤儿进程

## 4. 配置系统

### 问题

- 目前只有全局 `~/.fttm/config.yml`
- 无法按项目配置

### 解决方案

#### 4.1 分层配置

| 层级     | 文件位置                     | 优先级 |
| -------- | ---------------------------- | ------ |
| 系统级   | `~/.fttm/config.yml`         | 最低   |
| 项目级   | `<project>/.fttm/config.yml` | 中     |
| CLI 参数 | `--agent opencode`           | 最高   |

#### 4.2 配置 Schema

```yaml
# 支持的配置项
agent: claude | codex | rovodev | opencode
model: string # agent 默认 model
maxConsecutiveFailures: 3
preventSleep: true | false
maxRecentIterations: 10 # 新增
commitStrategy: all | failed-only | none # 新增
```

## 5. 观测性

### 问题

- token 消耗只在运行时显示，运行后无法追溯
- 没有运行报告

### 解决方案

#### 5.1 运行报告

运行结束后生成 `.fttm/runs/<run-id>/report.json`：

```json
{
  "runId": "run-xxx",
  "prompt": "reduce latency",
  "status": "completed | aborted | failed",
  "totalIterations": 15,
  "successfulIterations": 12,
  "failedIterations": 3,
  "totalTokens": {
    "input": 150000,
    "output": 80000,
    "cacheRead": 50000,
    "cacheCreation": 10000
  },
  "duration": "00:45:23",
  "commits": ["commit sha1", "commit sha2", ...],
  "abortedReason": null
}
```

#### 5.2 持久化 Stats

- 不只是 UI 显示，写入 `stats.jsonl`
- 支持后续分析/绘图

#### 5.3 JSON 输出模式

```sh
# 运行并输出 JSON 到文件
fttm "migrate auth" --json-out /tmp/run.json

# 运行并输出 JSONL 流（实时）
fttm "migrate auth" --json-stream

# 运行时实时输出 JSONL 到 stdout（便于管道到监控）
fttm "migrate auth" --json-stream --json-stdout
```

**JSON 流每行格式（每个迭代结束时输出）**：

```jsonl
{"event":"iteration_start","iteration":1,"timestamp":"2026-04-06T10:00:00Z"}
{"event":"iteration_end","iteration":1,"success":true,"summary":"...","duration_ms":45000,"tokens":{"input":5000,"output":3000}}
{"event":"iteration_start","iteration":2,"timestamp":"2026-04-06T10:00:45Z"}
{"event":"iteration_end","iteration":2,"success":false,"summary":"...","duration_ms":30000,"tokens":{"input":4000,"output":2000}}
{"event":"run_end","status":"completed","total_iterations":2,"duration_ms":75000,"commits":["abc123","def456"]}
```

**用途**：

- 接入监控系统（Grafana、Datadog）
- 自定义 dashboard
- CI/CD 集成
- 日志聚合

#### 5.4 Log 改进

- `--verbose` 输出更详细
- 结构化 log（JSON）便于程序解析

#### 5.5 进程监控

##### 5.5.1 PID 文件

运行启动时写入 `.fttm/runs/<run-id>/pid`：

```
12345
```

无论 foreground 还是 detach 都写，用于：

- crash recovery（知道哪个进程需要清理）
- `fttm status` 列出所有运行中的任务

##### 5.5.2 进程列表

```sh
fttm status
```

输出示例：

```
RUNNING:
  run-abc123  pid=12345  iter=5/∞   elapsed=00:23:15  status=running
  run-def456  pid=67890  iter=12/20  elapsed=01:05:30  status=waiting (backoff)

DETACHED:
  run-abc123  (pid 12345)
  run-def456  (pid 67890)
```

##### 5.5.3 扫描方式

不依赖 PID 文件，可扫描 `.fttm/runs/` 目录：

```sh
# 查找所有包含 pid 文件且进程仍在运行的 run
ls .fttm/runs/*/pid | while read pidfile; do
  pid=$(cat $pidfile)
  if kill -0 $pid 2>/dev/null; then
    runid=$(basename $(dirname $pidfile))
    echo "$runid pid=$pid"
  fi
done
```

##### 5.5.4 无需 detach 也可监控

- foreground 模式同样写 PID 文件
- `fttm status` 看到的是 `.fttm/runs/` 下所有活跃 run，不区分模式

##### 5.5.5 中央注册表（跨项目监控）

为支持全局 `fttm status`，使用中央注册表：

```
~/.fttm/runs-registry.json
```

```json
{
  "version": 1,
  "runs": [
    {
      "runId": "abc123",
      "project": "/path/to/project-a",
      "cwd": "/path/to/project-a",
      "pid": 12345,
      "startedAt": "2026-04-06T10:00:00Z",
      "status": "running",
      "currentIteration": 5,
      "maxIterations": null
    }
  ]
}
```

**写操作：乐观锁（不用文件锁）**

```
读 → 检查version → 改 → 写（原子覆盖）
```

如果写入时发现 version 不匹配，重新读再写。

**启动时清理孤儿**：

- 读取 registry
- 对每个 pid 发 `kill -0` 检查存活
- 死掉的 entry 移除或标记为 `stopped`
- 写回 registry

**心跳机制**（可选）：

- detach 进程每 60s 更新一次 `lastHeartbeat`
- 超过 120s 没心跳认为已死亡

### 问题

- 无法在迭代前后插入自定义逻辑
- 无法 hook 到 commit 前后

### 解决方案

#### 6.1 Hooks

```typescript
interface Hooks {
  beforeIteration?: (ctx: IterationContext) => void | Promise<void>;
  afterIteration?: (
    ctx: IterationContext,
    result: AgentResult,
  ) => void | Promise<void>;
  beforeCommit?: (
    ctx: IterationContext,
    changes: string[],
  ) => void | Promise<boolean>;
  afterCommit?: (ctx: IterationContext, sha: string) => void | Promise<void>;

```

- 独立 npm 包作为插件

## 8. 统筹 Agent（Coordinator）

### 问题

- 当前 agent 在 autonomy 模式下直接干活，不提问、不确认、不审查边界
- 危险操作（删除文件、修改系统配置、重写核心业务逻辑）没有人工审核
- 迭代之间没有协调者，agent 可能重复走弯路或忽略关键风险
- 正常交互式 agent 会问很多问题确认需求，但 fttm 的 autonomy 模式跳过了这一步

### 解决方案

#### 8.1 统筹 Agent 角色

引入一个 **Coordinator agent**（可配置为与 worker 不同的模型），负责：

| 职责         | 说明                                                                                 |
| ------------ | ------------------------------------------------------------------------------------ |
| **任务分解** | 把大目标拆成小的、可验证的 iteration scope                                           |
| **风险审查** | 在 agent 执行前，审查 scope 是否涉及危险操作（删除、重写、生产修改）                 |
| **进度指导** | 根据 notes.md 判断当前方向是否正确，给出指导性意见                                   |
| **提问转发** | 如果 worker agent 遇到不清晰的需求，向 Coordinator 报告，由 Coordinator 转给用户确认 |
| **质量把关** | 迭代结束后评估结果是否符合预期，决定是否需要重试                                     |

#### 8.2 交互模式

```
用户 → fttm "重构认证模块"
    ↓
Coordinator（审查 + 分解）→ "这个任务包含 3 个子目标，我先确认：1) 现有认证流程是什么？2) 改成 JWT 方案？"
    ↓
用户确认需求
    ↓
Coordinator 分解任务 → worker agent 执行 iteration
```

**两种运行模式**：

| 模式                   | 说明                                                |
| ---------------------- | --------------------------------------------------- |
| `--autonomous`（默认） | Worker 直接干，不提问                               |
| `--supervised`         | Coordinator 监督，危险操作需确认                    |
| `--interactive`        | 用户直接和 agent 对话（类似标准 claude/codex 交互） |

#### 8.3 Coordinator 配置

```yaml
# .fttm/config.yml
coordinator:
  enabled: true
  agent: claude # 可用不同模型做 coordinator
  model: opus
  askOnDanger: true # 危险操作前暂停
  approveOnReview: false # 迭代结果是否需要 coordinator 确认
```

#### 8.4 与现有单 agent 的关系

- **向后兼容**：`--agent single`（默认）保持当前行为
- **新增**：`--agent supervised` 启用 Coordinator 监督模式
- `fttm start "task" --supervised` → Coordinator 审查后转给 worker

#### 8.5 危险操作检测

Worker agent 执行前，Coordinator 扫描 scope 关键词：

| 风险等级 | 关键词示例                             | 行为                      |
| -------- | -------------------------------------- | ------------------------- |
| 低       | `docs/`、`tests/`、`*.md`              | 直接执行                  |
| 中       | `src/`、`*.ts`、`*.py`                 | Coordinator review 后执行 |
| 高       | `rm -rf`、`DROP TABLE`、`.env`、`.key` | 必须用户确认              |
| 极高     | `sudo`、`chmod 777`、生产环境配置      | 拒绝执行，要求降级        |
