# fttm v0.2.1 架构重构规格文档

**版本**: 0.2.1  
**目标**: 全面优化架构，解决记忆膨胀、观测性、配置系统等问题

---

## v0.1 vs v0.2.1 对比

| 特性               | v0.1                        | v0.2.1                                        |
| ------------------ | --------------------------- | --------------------------------------------- |
| **记忆系统**       | notes.md 无限膨胀           | 滑动窗口 + 分层记忆                           |
| **记忆注入**       | 全部历史注入 prompt         | 按需读取 + 近期完整注入                       |
| **配置**           | 仅全局 `~/.fttm/config.yml` | 分层配置（系统/项目/CLI）                     |
| **运行报告**       | 无                          | `report.json` 持久化                          |
| **Token 统计**     | 仅运行时显示                | `stats.jsonl` 持久化                          |
| **Checkpoint**     | 无                          | 迭代快照 + Crash 恢复                         |
| **CLI**            | 单命令                      | 子命令（start/resume/status/log/report/stop） |
| **Hooks**          | 无                          | 迭代前后/提交前后钩子                         |
| **目录结构**       | `.fttm/runs/<id>/` 扁平     | `.fttm/runs/<id>/iterations/` 分层            |
| **Agent 能力声明** | 无                          | `AgentMetadata` 接口                          |

### 破坏性变更

| 变更                     | v0.1     | v0.2.1                          |
| ------------------------ | -------- | ------------------------------- |
| `.fttm/` 目录            | 扁平结构 | 子目录 `iterations/`            |
| notes.md                 | 完整历史 | 仅近期 + 归档                   |
| CLI 子命令               | 无       | `fttm status`/`log`/`report` 等 |
| `maxConsecutiveFailures` | 硬编码 3 | 可配置                          |

## 1. 记忆与上下文系统

### 问题

- `notes.md` 无限膨胀，50+ 迭代后 context 消耗大
- 所有历史完整注入 prompt，导致 token 浪费

### 解决方案

#### 1.1 滑动窗口记忆

- 只保留最近 **N 次迭代**的完整内容（默认 N=10）
- 早期迭代合并成摘要
- 配置项：`maxRecentIterations: 10`

#### 1.2 分层记忆

| 层级              | 内容                               | 注入方式     |
| ----------------- | ---------------------------------- | ------------ |
| 近期（最近 N 次） | 完整 summary + changes + learnings | 注入 prompt  |
| 中期              | 仅 summary（压缩成一行）           | 注入 prompt  |
| 历史              | JSONL 原始文件                     | agent 自己读 |

#### 1.3 按需读取

- prompt 里告诉 agent："如需了解历史详情，读取 `.fttm/runs/<run-id>/iteration-K.jsonl`"
- agent 自己决定是否需要查历史，而非每次都注入

## 2. Agent 抽象层

### 问题

- Agent 接口虽已抽象，但扩展新 agent 仍需修改多处
- 不同 agent 的 token 计数逻辑分散

### 解决方案

#### 2.1 统一 Token 报告接口

```typescript
interface AgentResult {
  output: AgentOutput;
  usage: {
    inputTokens: number;
    outputTokens: number;
    cacheReadTokens?: number; // claude 特有
    cacheCreationTokens?: number; // claude 特有
  };
}
```

#### 2.2 Agent 注册表

```typescript
const AGENTS = {
  claude: ClaudeAgent,
  codex: CodexAgent,
  rovodev: RovoDevAgent,
  opencode: OpenCodeAgent,
} as const;
```

#### 2.3 Agent 能力声明

```typescript
interface AgentMetadata {
  name: string;
  supportsStreaming: boolean;
  supportsJsonSchema: boolean;
  supportsCustomModel: boolean;
}
```

## 3. 状态管理与恢复

### 问题

- crash 后只能 resume，可能丢失最后一次迭代
- 没有手动 checkpoint

### 解决方案

#### 3.1 迭代快照

每次迭代开始前写入 `.fttm/runs/<run-id>/checkpoint.json`：

```json
{
  "iteration": 5,
  "startedAt": "ISO8601",
  "workingTreeHash": "sha"
}
```

#### 3.2 自动恢复

- 启动时检测 checkpoint
- 如果 checkpoint 显示上次未完成，提供选项：
  - 继续（retry last iteration）
  - 跳过（treat as failed）
  - 中止

#### 3.3 Crash 保护

- 捕获 SIGTERM/SIGINT 时先写 checkpoint 再退出
- 防止孤儿进程

## 4. 配置系统

### 问题

- 目前只有全局 `~/.fttm/config.yml`
- 无法按项目配置

### 解决方案

#### 4.1 分层配置

| 层级     | 文件位置                     | 优先级 |
| -------- | ---------------------------- | ------ |
| 系统级   | `~/.fttm/config.yml`         | 最低   |
| 项目级   | `<project>/.fttm/config.yml` | 中     |
| CLI 参数 | `--agent opencode`           | 最高   |

#### 4.2 配置 Schema

```yaml
# 支持的配置项
agent: claude | codex | rovodev | opencode
model: string # agent 默认 model
maxConsecutiveFailures: 3
preventSleep: true | false
maxRecentIterations: 10 # 新增
commitStrategy: all | failed-only | none # 新增
```

## 5. 观测性

### 问题

- token 消耗只在运行时显示，运行后无法追溯
- 没有运行报告

### 解决方案

#### 5.1 运行报告

运行结束后生成 `.fttm/runs/<run-id>/report.json`：

```json
{
  "runId": "run-xxx",
  "prompt": "reduce latency",
  "status": "completed | aborted | failed",
  "totalIterations": 15,
  "successfulIterations": 12,
  "failedIterations": 3,
  "totalTokens": {
    "input": 150000,
    "output": 80000,
    "cacheRead": 50000,
    "cacheCreation": 10000
  },
  "duration": "00:45:23",
  "commits": ["commit sha1", "commit sha2", ...],
  "abortedReason": null
}
```

#### 5.2 持久化 Stats

- 不只是 UI 显示，写入 `stats.jsonl`
- 支持后续分析/绘图

#### 5.3 JSON 输出模式

```sh
# 运行并输出 JSON 到文件
fttm "migrate auth" --json-out /tmp/run.json

# 运行并输出 JSONL 流（实时）
fttm "migrate auth" --json-stream

# 运行时实时输出 JSONL 到 stdout（便于管道到监控）
fttm "migrate auth" --json-stream --json-stdout
```

**JSON 流每行格式（每个迭代结束时输出）**：

```jsonl
{"event":"iteration_start","iteration":1,"timestamp":"2026-04-06T10:00:00Z"}
{"event":"iteration_end","iteration":1,"success":true,"summary":"...","duration_ms":45000,"tokens":{"input":5000,"output":3000}}
{"event":"iteration_start","iteration":2,"timestamp":"2026-04-06T10:00:45Z"}
{"event":"iteration_end","iteration":2,"success":false,"summary":"...","duration_ms":30000,"tokens":{"input":4000,"output":2000}}
{"event":"run_end","status":"completed","total_iterations":2,"duration_ms":75000,"commits":["abc123","def456"]}
```

**用途**：

- 接入监控系统（Grafana、Datadog）
- 自定义 dashboard
- CI/CD 集成
- 日志聚合

#### 5.4 Log 改进

- `--verbose` 输出更详细
- 结构化 log（JSON）便于程序解析

## 6. 扩展性

### 问题

- 无法在迭代前后插入自定义逻辑
- 无法 hook 到 commit 前后

### 解决方案

#### 6.1 Hooks

```typescript
interface Hooks {
  beforeIteration?: (ctx: IterationContext) => void | Promise<void>;
  afterIteration?: (
    ctx: IterationContext,
    result: AgentResult,
  ) => void | Promise<void>;
  beforeCommit?: (
    ctx: IterationContext,
    changes: string[],
  ) => void | Promise<boolean>;
  afterCommit?: (ctx: IterationContext, sha: string) => void | Promise<void>;
}
```

#### 6.2 插件系统（v0.3 考虑）

- 独立 npm 包作为插件
- `~/.fttm/plugins/` 目录加载

## 7. CLI 改进

### 解决方案

#### 7.1 子命令

```sh
fttm start "<prompt>"        # 新建运行
fttm resume                  # 恢复运行
fttm status                  # 查看当前状态
fttm log                     # 查看运行日志
fttm report                  # 查看运行报告
fttm stop                    # 停止当前运行
```

#### 7.2 `--dry-run`

- 不真正执行，只生成 iteration prompt
- 便于调试

#### 7.3 `--verbose` / `--quiet`

- 控制输出详细程度

## 8. 目录结构变化

### 当前结构

```
.fttm/
└── runs/
    └── <run-id>/
        ├── prompt.md
        ├── notes.md
        ├── base-commit
        ├── output-schema.json
        └── iteration-N.jsonl
```

### 目标结构

```
.fttm/
├── config.yml           # 项目级配置（可选）
└── runs/
    └── <run-id>/
        ├── manifest.json      # 新的：run 元信息
        ├── prompt.md
        ├── notes.md           # 压缩后的近期记忆
        ├── notes-archive.md   # 新的：历史摘要归档
        ├── report.json        # 新的：运行报告
        ├── stats.jsonl        # 新的：token 统计
        ├── checkpoint.json    # 新的：迭代快照
        ├── base-commit
        ├── output-schema.json
        └── iterations/
            └── iteration-N.jsonl  # 移动到子目录
```
